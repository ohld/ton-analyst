# Professionalization Plan

Goal: keep `ton-analyst` compact, but make it feel like a professional skill
product: predictable installs, small hot context, strong validation, and clear
operating rules.

## Principles

- **Module:** one durable capability with a clear Interface and Implementation.
- **Depth:** keep the visible skill Interface small; move detailed behaviour into
  references, scripts, tests, and docs.
- **Seam:** make runtime variation explicit: Claude Code vs Codex, Dune MCP vs
  fallback, TonAPI vs label cache.
- **Adapter:** use thin command/install/update wrappers at each Seam.
- **Leverage:** one convention or test should protect many future analyses.
- **Locality:** bugs, terminology, and release rules should have one obvious home.

## P0 — Already Shipped

- Merge useful open PRs and close stale queue.
- Add built-in update check and post-merge version bumping.
- Fix auto-bump so it does not depend on commit message text.
- Run CLI pytest suite in CI.
- Reject tracked cache artifacts in CI.
- Remove duplicate stale workflow and generated bytecode.
- Refresh README structure and local install docs.

## P1 — Skill Interface Depth

1. Add `skills/ton-analyst/reference/index.md`.
   Map user intents to exact references: wallet investigation, supply, DeFi,
   CEX flows, NFT, jettons, staking, priority mining, dashboards, reports.

2. Shrink `SKILL.md` into a router-style Interface.
   Keep trigger, update check, hard conventions, and routing. Move detailed CLI,
   report, table, and external-resource text behind references.

3. Add `skills/ton-analyst/reference/report-format.md`.
   Include query ids, SQL files, result snapshots, table list, date range,
   filters, excluded entities, confidence, blind spots, and manual checks.

4. Add `CONTEXT.md`.
   Define Account, Wallet, Address, Message, Transaction, Jetton, Label, Entity,
   Balance Change Log, Snapshot, and common rejected synonyms.

5. Add `docs/adr/`.
   Record durable conventions: percentage ratios stay fractional, Tonviewer links
   are mandatory, CLI before raw TONAPI for single-address work, and
   `balances_history` is a change log.

## P2 — Professional Support Surface

1. Add `ton doctor --json`.
   Check skill version, update status, wrapper path, Python env, Dune key,
   TONAPI key, label cache, reference files, and CLI health.

2. Add JSON mode to `ton-analyst-update-check`.
   Keep current text marker for skill prompt use; expose structured output for
   doctor, tests, and future automation.

3. Harden update check.
   Add snooze/backoff, just-updated marker, clearer cache state, and stronger
   semver comparison tests.

4. Unify setup.
   Let root `./setup` optionally install both the skill symlink and the `ton` CLI
   wrapper, so local users have one install Seam.

## P3 — CLI Architecture

1. Split `bin/ton` into an importable `ton_analyst_cli` package.
   Keep `bin/ton` as a thin Adapter.

2. Create operation contracts.
   Define operations like `acc`, `tx`, `doctor`, and future `label` behind stable
   Interfaces that CLI/tests can call directly.

3. Create provider Adapters.
   Separate TonAPI HTTP, label cache, output formatting, and pruning into modules
   with focused tests.

4. Keep no-network test defaults.
   Continue using mocked HTTP and fixture-based tests so CI remains fast and
   deterministic.

## P4 — Skill Quality Gates

1. Add prompt/reference lint.
   Validate local links, referenced files, SQL example presence, table-schema
   coverage, and `SKILL.md` target line count.

2. Add golden routing fixtures.
   For each user question, assert chosen references, expected SQL shape, and
   required caveats.

3. Add `ARCHITECTURE.md`, `AGENTS.md`, and `llms.txt`.
   Give humans and agents a clear read order and explain the hot-context vs
   reference split.

4. Improve changelog format.
   Use `Impact`, `Interface changes`, `Migration`, and `Verification` sections
   for releases that affect users or agent behaviour.

5. Add local query learnings.
   Store Dune quirks, expensive joins, label corrections, and report caveats in a
   small append-only local memory file.
